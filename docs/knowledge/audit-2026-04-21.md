# Аудит базы знаний (актуально на 21.04.2026)

## Что реализовано (кратко)

- **Модели в БД** (Prisma): `KnowledgeBase`, `KnowledgeItem` (FILE/TEXT/URL), `Document`, `Chunk` (+ `embedding vector(1536)` через pgvector, + generated `content_tsv` с GIN), `EmbeddingJob`, `SourceLink`, связь `AssistantKnowledgeBase`.
- **Импорт файлов** (`lib/knowledge-ingest.ts`): PDF (pdf-parse), DOCX (mammoth), TXT/MD/JSON; SHA‑256 хеш файла; идемпотентность по `metadata.sourceUrl`/`contentSha256`; чанкинг (≈1800 симв., overlap 200, paragraph-aware); запись `KnowledgeItem`+`Document`+`Chunk` в транзакции; постановка `EmbeddingJob` (QUEUED).
- **Длинный TEXT**: выше `TEXT_CHUNK_THRESHOLD=2200` сохраняется так же, как файл — `Document`+`Chunk`+`EmbeddingJob`. Есть API‑лимит `MAX_KNOWLEDGE_TEXT_CHARS=500_000`.
- **URL** (`lib/url-ingest.ts`): `fetch` с таймаутом 18 с и лимитом 2 МБ, очистка HTML от `<script>/<style>/<tags>`, чанкинг, создание `Document` для внутреннего объекта `url/<tenant>/<itemId>.txt`, постановка эмбеддингов.
- **RAG-сбор контекста** (`lib/knowledge-context.ts`): гибрид — FTS (`to_tsvector('simple')` + `plainto_tsquery`), ILIKE fallback, свежие чанки по `updatedAt`, семантика через pgvector (`<=>`, косинус) при наличии ключа эмбеддингов; плюс «голые» TEXT/URL без документа. Слияние по score, обрезка до `maxChars`.
- **Промпт «только факты»** (`lib/knowledge-grounding.ts`) — общие правила грунтинга добавляются в системный промпт и для ассистентов, и для агентов. Агенты теперь подгружают связанные `knowledgeLinks` так же, как ассистенты.
- **Эмбеддинги**: `lib/tenant-embedding-config.ts` (OpenRouter → OpenAI fallback), `lib/embeddings-api.ts` (OpenAI-совместимый `POST /embeddings`), воркер `lib/knowledge-embed-worker.ts` + маршрут `POST /api/knowledge/embeddings/worker` (защищён заголовком `x-embedding-worker-secret`).
- **ИИ-подсказки** (`lib/knowledge-suggest-ai.ts`, `POST /api/knowledge/[id]/suggest-split`): черновик → JSON-массив `{title, content}` для ручной вставки; **не пишет** в БД автоматически.
- **UI** (`components/knowledge-page-client.tsx`): студия — список/поиск баз, вкладки «Файлы → RAG», «Текст», «URL», таблица материалов с типом/заголовком/числом чанков/статусом, кнопка ИИ-подсказок, ссылки на `/assistants` и `/docs/knowledge`.
- **Документация**: `/docs/knowledge` (страница в приложении), `docs/knowledge/limitations-and-mitigations.md`, `docs/infra/deploy-and-tls-checklist.md` (раздел про pgvector и cron).
- **Инфра**: миграция `20260221140000_knowledge_chunk_vectors_fts` применена на prod, `postgresql-16-pgvector` установлен, cron `*/2 * * * *` активен, корневой `.env` авто-грузится Next через `apps/web/scripts/start-with-root-env.js`.

## Потенциальные риски и зоны улучшения

1. **Извлечение текста — только синхронно на веб-процессе.** Крупные PDF/DOCX блокируют event loop и держат память. Решение: перенести `extractTextFromFile` в отдельный воркер (например, BullMQ или очередь на Redis + отдельный процесс pm2), а API сразу отвечает `QUEUED`.
2. **Нет OCR.** Сканы PDF и изображения игнорируются. Решение: опциональный путь через Tesseract/OCR-сервис по кнопке «распознать» (явная цена времени и CPU).
3. **URL без headless-браузера.** SPA/антибот → пустой текст. Решение: отдельный воркер с Playwright (или внешний сервис типа Browserless) и очередь с лимитом на домен; соблюдать `robots.txt` и `User-Agent` политики сайта.
4. **Чанкинг по символам, без учёта структуры.** Для Markdown/HTML лучше бить по заголовкам/секциям, сохраняя «путь» (breadcrumbs) в `metadata` чанка — это заметно улучшает точность RAG и цитирование.
5. **Ранжирование FTS+вектор — простое суммирование баллов.** Рекомендуется более честный **RRF (reciprocal rank fusion)** или нормализация косинуса/`ts_rank_cd` в [0..1] перед сложением.
6. **Нет перезапроса/реранкера.** Для сложных вопросов помогает CrossEncoder/`rerank` (есть у Cohere/OpenRouter-совместимых); можно опционально включить для top‑50 → top‑10.
7. **Цитаты в ответах.** Сейчас модель может не ссылаться на фрагменты. Решение: добавлять к чанку маркер `⟨#1⟩… ⟨#2⟩…` и требовать в промпте ссылки вида `[#1]`; фронт умеет подсвечивать. Это резко снижает галлюцинации и упрощает проверку.
8. **Токен-бюджет контекста статический (`DEFAULT_MAX=12_000`).** Для разных моделей нужно выставлять его по модели ассистента (например, GPT‑4o‑mini vs Claude Sonnet). Добавить в `settings` модели `maxContextChars`.
9. **Дедупликация — только для файлов.** У TEXT/URL нет хеша — при повторной загрузке одной и той же страницы будут дубли чанков. Решение: SHA‑256 по итоговому извлечённому тексту + `ON CONFLICT` по `(tenantId, knowledgeBaseId, contentSha256)`.
10. **Метрики.** Сейчас время ingest пишется в `metadata`. Полезно завести агрегаты: число чанков/базу, доля с эмбеддингами, 95‑перцентиль времени ингеста — отдельная админ-страница `/knowledge/health` с SQL-агрегациями.
11. **RAG-audit по диалогу.** Сейчас не сохраняется, какие чанки ушли модели. Решение: писать в `Message.metadata` список `chunkIds`, чтобы администратор мог открыть «что модель видела», это сильно ускоряет разбор некорректных ответов.
12. **Удаление материала оставляет «дыры» в `EmbeddingJob`.** `Chunk` удаляются каскадом, а «осиротевшие» QUEUED-задачи можно чистить cron‑скриптом `UPDATE EmbeddingJob SET status='FAILED' WHERE status='QUEUED' AND updatedAt < now() - interval '1 day';`.
13. **Лимит размера URL-ответа = 2 МБ.** Для длинных статей может быть мало. Сделать настройкой тенанта.
14. **`KnowledgeItem.content` режется до 8000 симв.** Это сознательно, но для UI поиска полезно сохранять «preview» отдельно и не подмешивать этот текст в контекст (чтобы не дублировать чанки). Сейчас такого дублирования нет, но стоит явно задокументировать в коде.
15. **Безопасность URL-ингеста.** Стоит запретить частные сети (10/8, 192.168, 169.254, 127/8, IPv6 linklocal) в `url-ingest.ts`, чтобы не было SSRF.
16. **Лимиты на запись.** На больших объёмах полезно квотирование по тенанту: максимум баз/материалов/чанков, чтобы не раздуть таблицы.
