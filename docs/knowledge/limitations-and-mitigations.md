# База знаний: ограничения и как их снимать

## 1. Качество RAG зависит от наполнения и модели; промпт не гарантирует 100% правду

**Почему:** языковая модель всегда может «додумать» при слабом контексте, узком чанкинге или противоречивых материалах.

**Что делать:**

- **Структура базы:** одна база — одна тема; избегайте смешения маркетинга и юридических фактов в одной куче.
- **Чанкинг:** держите фрагменты цельными (мы режем по перекрытию и границам абзацев); при импорте огромных PDF проверяйте, что текст не «рваный» (скан без OCR).
- **Подбор:** после миграции pgvector включайте воркер эмбеддингов и cron; гибрид FTS + вектор + свежие чанки уже в коде — не отключайте эмбеддинги без причины.
- **Промпт ассистента:** дополнительно в системном промпте можно запретить цифры/даты без цитаты из материалов (общие правила уже подмешиваются из `knowledge-grounding.ts`).
- **Модель:** для «строго по тексту» чаще лучше модель посвежее/помощнее и температура **0.2–0.5** в настройках провайдера (если поддерживаете отдельно для чата).
- **Оценка:** периодически прогоняйте контрольные вопросы с известными ответами из базы и правьте материалы.

## 2. URL: нет полноценного robots.txt; SPA и антибот дают пустой текст

**Почему:** текущий пайплайн — один HTTP GET, очистка HTML, без браузера и без разбора `robots.txt`.

**Что делать:**

- Для **важных** страниц: сохраняйте как **PDF** или копируйте текст вручную в базу.
- **Партнёрские API / экспорт** (Notion, Confluence) — надёжнее, чем «голый» URL (можно вынести в отдельный коннектор позже).
- Если нужен **robots + рендер JS**: отдельный сервис (Playwright/Puppeteer в воркере, очередь задач, лимиты по домену) — осознанно дороже по CPU и поддержке.
- **Правовое:** не обходите paywall и не нарушайте ToS сайта-источника.

## 3. Миграции Prisma раньше были в `.gitignore`

**Почему:** без SQL в репозитории прод не получает схему после `git pull`.

**Что сделано:** строка `packages/db/prisma/migrations` **удалена** из `.gitignore`; миграции коммитятся обычным `git add`.

**На уже задеплоенных серверах:** один раз установить пакет `postgresql-16-pgvector` (или аналог для вашей версии PG), создать расширение суперпользователем БД:

```bash
sudo -u postgres psql -d ИМЯ_БД -c 'CREATE EXTENSION IF NOT EXISTS vector;'
```

затем `npx prisma migrate deploy` из корня репозитория с загруженным `.env` (чтобы был `DATABASE_URL`).

## 4. Cron воркера эмбеддингов

Скрипт: `infra/scripts/embedding-worker-cron.sh` (переменные `AI_SITE_AL_ROOT`, `AI_SITE_AL_PORT` при необходимости). Файл должен быть с **LF**-окончаниями строк (в репозитории задано в `.gitattributes` для `*.sh`).

Пример crontab (раз в 2 минуты):

```cron
*/2 * * * * /var/www/ai.site-al.ru/infra/scripts/embedding-worker-cron.sh
```

Лог по умолчанию: `/var/log/ai-site-al-embedding-worker.log`.

После изменения `.env` перезапустите процесс Next.js (`next start`), чтобы подтянулись новые переменные (если не используете внешнюю подстановку env в systemd/pm2).

### Next.js в монорепо (`npm run start -w apps/web`)

Next по умолчанию читает `.env*` только из своего каталога (`apps/web`). Чтобы не дублировать файлы, используется загрузчик `apps/web/scripts/start-with-root-env.js` (подключён как `npm start`), который подхватывает корневые `.env`, `.env.local`, `.env.production`, `.env.production.local` и экспортирует их в процесс Next.

Итог: **единственный `.env` — в корне** `/var/www/ai.site-al.ru/.env` (правится по SSH, в git не попадает). Никаких копий в `apps/web/.env*` держать не нужно и не следует.
